### 2.3 主路径（variant path）默认改为隔离输出

本周在主路径上做了一个关键工程决策：**默认不覆盖既有 disease 结果目录**。

此前变异驱动路径和标准 disease 路径共用 `<out_dir>/<disease_slug>/`，这会带来两个问题：

1. 不同病人的 variant 运行容易互相覆盖；  
2. 已完成的标准路径结果（例如当前 breast 标准路径）容易和主路径遗留产物混在一起。  

本周改为：

- 当 `variant_analysis.enabled: true` 时，默认输出到  
  `<pipeline.out_dir>/<disease_slug>/variant_runs/<sample_id>/<run_id>/`

其中：

- `sample_id` 优先取 `variant_analysis.sample_id`；
- 若未显式设置，则从 VCF / FASTQ 文件名推断；
- `run_id` 默认是启动时间戳，也可以显式覆盖。

这一改动直接解决了“同 disease 多病人结果冲突”的问题，也为后续接入真实病人样本做好了路径层面的准备。

### 2.4 主路径默认切到 `singularity`，匹配 AutoDL / HPC 场景

考虑到 AutoDL / HPC 环境下 Docker 往往不可用，本周将默认 Sarek profile 改为：

- `singularity`

同时在配置注释中保留：

- 本地开发 / 具备 Docker 权限的机器可改回 `docker`

这项改动降低了主路径真实落地时的环境阻力，避免“文档支持 but 默认配置不支持”的割裂。

其中包含：

### 2.7 真实数据入口方向已经明确：优先走 GDC 的 VCF 路线

本周还推进了真实数据入口选择。当前判断是：

- 真实数据第一阶段优先走 **VCF 路线**，而不是直接上 FASTQ；
- 原因在于 GDC 的原始 WXS 读段多数为 controlled access，获取门槛更高；
- 相比之下，当前已经能在 GDC `Repository` 中稳定筛到：
  - `Project ID = TCGA-BRCA`
  - `Experimental Strategy = WXS`
  - `Data Category = Simple Nucleotide Variation`
  - `Data Format = VCF`

当前可见文件多为：

- `mutect2_raw_somatic_mutation.vcf.gz`
- `muse_raw_somatic_mutation.vcf.gz`
- `varscan2_raw_somatic_mutation.vcf.gz`
- `pindel_raw_somatic_mutation.vcf.gz`

下一步最合理的真实验证顺序是：

1. 优先选择一个 `mutect2_raw_somatic_mutation.vcf.gz`；  
2. 检查 header 是否已有 `CSQ` / VEP 注释；  
3. 若无，则补做 VEP 注释；  
4. 再喂给主路径。

### 3.2 GDC 的“可见文件”不等于“可直接下载且可直接使用”

需要明确两点：

1. 当前看到的 `raw_somatic_mutation.vcf.gz` 不一定带 VEP 注释；  
2. GDC 中很多原始 WXS 或配套数据受 controlled access 约束。  

因此接下来真实数据验证的阻塞点不在“主路径代码能不能跑”，而在：

- 能否拿到一个允许使用的 BRCA / PIK3CA VCF；
- 该 VCF 是否需要二次注释。

### 3.3 主路径与标准路径虽然已经分离，但历史遗留仍需谨慎归档

本周已将历史 variant 遗留目录从当前 breast 标准路径根目录中移出，放到：

- `breast_cancer/_archived_mainpath/variant_analysis`

这一步避免了未来把标准路径产物误当成主路径新结果。  
但后续仍建议补一个更正式的：

- `run_manifest.json`
- 或统一 index / archive 方案

让多轮运行的归档可机器读取，而不仅靠目录名称判断。
