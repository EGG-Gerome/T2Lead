## 二、OpenMM `CUDA_ERROR_UNSUPPORTED_PTX_VERSION` 与「12.8 / 13」的关系

### 2.1 为何 12.8 仍可能「不行」（针对的是 OpenMM，不是 PyTorch）

- PyTorch `+cu128` 构建通常会为新一代 GPU 提供**可用的**预编译内核或兼容的 JIT 路径，所以在 RTX 5090 上训练/推理可以正常。
- OpenMM 报错 `UNSUPPORTED_PTX_VERSION (222)` 发生在 **OpenMM 加载自己的 CUDA 模块**时：其编译进包的 PTX **过旧**，无法在目标 GPU 上完成 JIT 到 sm_120 的有效加载，是 **OpenMM 包与 Blackwell 的匹配问题**。

### 2.2 代码侧采取的措施（本周）

在 `src/drugpipe/lead_optimization/md_simulation.py` 中增加 **平台可用性实测**：在选用 CUDA 前运行微型上下文；若加载失败则 **自动回退 CPU**，避免静默失败或整段 MM-GBSA 全空。

**验证情况**：在开发环境中对 **CPU 路径**做过小规模能量测试（如小分子配体单体系），确认 OpenMM 在 CPU 上可完成最小化与能量读取；**大蛋白复合物**在 CPU 上可行但耗时大，**尚未**作为生产跑数在本周报周期内完整跑满 10 条先导的 MM-GBSA 全量任务。

### 2.3 最新复跑校验（2026-03-27 15:28 日志）

在最新全链路日志 `20260327_152846_breast_cancer_full.log` 中，OpenMM 已可在本机正常选择 CUDA 平台（`OpenMM auto-selected platform: CUDA`），未再出现本周早期的 `UNSUPPORTED_PTX_VERSION` 作为首个阻塞点。  
当前 Stage 4 的主阻塞已转移为 **MM-GBSA 代码路径的能量类型兼容问题**：

- 典型报错：`'float' object has no attribute 'value_in_unit'`
- 影响：10/10 分子均在 complex 能量读取处失败，`complex=None`，导致 `MD simulation complete: 0 / 10`。

## 三、REINVENT4 子进程问题：原因与修复范围

### 3.1 现象（3 月 20 日日志）

- `reinvent` 返回非 0；日志中曾截断 traceback，易误判为「单一原因」。

### 3.2 已定位的两类问题

1. **动态库**：子进程未优先使用 conda 的 `libstdc++`，导致 RDKit 报 `GLIBCXX_3.4.31` 缺失。修复：在 `reinvent_bridge.py` 的 `subprocess.run` 中注入 `LD_LIBRARY_PATH`（conda env 的 `lib` 前置）。
2. **配置格式**：桥接生成的 TOML 与 **REINVENT 4.7.x** 的 schema 不一致（例如 `run_type` 位置、`[[stage]]`、评分组件结构、`TanimotoSimilarity` 的 `use_features` 等）。修复：按官方示例结构调整模板；ExternalProcess 评分脚本改为符合插件期望的 **JSON payload** 输出。

### 3.3 「RL 训练循环已验证」的准确表述

| 验证项                                                       | 状态                                                         |
| ------------------------------------------------------------ | ------------------------------------------------------------ |
| `reinvent --help` / 进程能 import RDKit 并进入主程序         | **已验证**（在注入 `LD_LIBRARY_PATH` 后）                    |
| 配置通过 Pydantic，进入 `staged_learning` 并创建输出 CSV 前缀文件 | **已验证**（短跑、CPU、限时场景下）                          |
| **完整 `max_steps`、生成稳定可读的 `reinvent4_1.csv` 分子列表、并接回 T2Lead 下游** | **桥接模块的生产参数全链路**：仍待单独验收；**极简 smoke**（见附录 A）已在本机 **exit 0** 跑完并写出 `reinvent4_1.csv`（含表头 + 多行 step 记录）。 |

**结论**：可表述为 **「在注入 conda `LD_LIBRARY_PATH` + REINVENT 4.7.x 格式 TOML 的前提下，CPU 上可完成短 RL 跑数并落盘 CSV」**；生产级 `n_steps`/batch 与 ExternalProcess 评分仍建议再跑一轮计时与结果抽检。

### 5.2 修复 MD 系统性失败（已定位根因，下周完成验收）

日志告警：

- `Complex GB energy failed for '...': 'float' object has no attribute 'value_in_unit'`

已定位为 `md_simulation.py` 中能量返回值类型兼容问题：同一调用链中有时返回 OpenMM `Quantity`，有时已是 `float`；当前代码对返回值统一按 `value_in_unit` 访问，触发属性错误并导致分子在同一位置系统性失败。  
下周动作：

1. 对能量读取做统一适配（`Quantity` 与 `float` 双分支）。
2. 为该路径补最小回归测试，防止后续重构再次引入同类 bug。
3. 复跑 top-N MD，确认不再出现“同一异常批量失败”。

### 5.3 肿瘤/正常双样本变异检测起步（5GB + 5GB FASTQ）

目标：把上游 DNA 变异信息稳定接入 T2Lead，实现“肿瘤突变驱动的靶点与结构更新”闭环。

1. **比对（BWA-MEM）**：肿瘤外显子 FASTQ、血液正常 FASTQ 分别比对至 GRCh38，产出 BAM。
2. **清洗与排序（SAMtools）**：坐标排序、去重与基础质控，得到可用于体细胞突变检测的标准化 BAM。
3. **体细胞突变检测（GATK Mutect2）**：联合肿瘤/正常样本，过滤背景变异，产出 VCF。
4. **变异注释（VEP / SnpEff，关键“3.5 步”）**：把 VCF 转为可解释的蛋白层变异（如 EGFR L858R）与突变氨基酸序列信息。
5. **回接 T2Lead（闭环）**：
    - 突变序列输入 ESMFold 生成突变蛋白结构（PDB）；
    - 进入 Stage 4（Docking + MD）评估候选分子对突变体的结合稳定性；
    - 形成“基因变异 -> 结构变化 -> 分子筛选”的端到端流程。

### 5.4 现有工程回归

1. 用生产级 `n_steps`/batch 再跑 REINVENT4，确认 `reinvent_bridge._parse_output` 与 Stage 3 衔接无回归（附录 A 已覆盖极简闭环）。
