## 二、OpenMM `CUDA_ERROR_UNSUPPORTED_PTX_VERSION` 与「12.8 / 13」的关系

### 2.1 几个容易混在一起的数字

| 来源 | 含义 |
|---|---|
| `torch 2.10.0+cu128` | PyTorch **随包编译**所针对的 CUDA toolkit 版本（12.8），用于 PyTorch 自己的 CUDA 算子。 |
| `nvidia-smi` 里的 **CUDA Version: 13.0**（示例） | 驱动**支持的最高 CUDA 运行时版本**，不等于你当前每个库实际用的 toolkit 版本。 |
| OpenMM（conda 安装的 8.4） | **独立**于 PyTorch 的另一套 CUDA/PTX 产物；其自带的 GPU 内核与 PTX 由 OpenMM 发布时决定。 |

因此：**你使用 PyTorch CUDA 12.8 没有问题**；驱动报 13.0 也不表示「必须用 13 才能跑 Blackwell」。问题在于 **OpenMM 这一份二进制里的 PTX / 架构覆盖** 是否包含 **sm_120（Blackwell）** 所需内容。

### 2.2 为何 12.8 仍可能「不行」（针对的是 OpenMM，不是 PyTorch）

- PyTorch `+cu128` 构建通常会为新一代 GPU 提供**可用的**预编译内核或兼容的 JIT 路径，所以在 RTX 5090 上训练/推理可以正常。
- OpenMM 报错 `UNSUPPORTED_PTX_VERSION (222)` 发生在 **OpenMM 加载自己的 CUDA 模块**时：其编译进包的 PTX **过旧**，无法在目标 GPU 上完成 JIT 到 sm_120 的有效加载。这与「系统里 PyTorch 是 12.8 还是驱动显示 13」**不是同一层问题**——是 **OpenMM 包与 Blackwell 的匹配问题**，不是「你没装 13 所以不行」。

### 2.3 代码侧采取的措施（本周）

在 `src/drugpipe/lead_optimization/md_simulation.py` 中增加 **平台可用性实测**：在选用 CUDA 前运行微型上下文；若加载失败则 **自动回退 CPU**，避免静默失败或整段 MM-GBSA 全空。

**验证情况**：在开发环境中对 **CPU 路径**做过小规模能量测试（如小分子配体单体系），确认 OpenMM 在 CPU 上可完成最小化与能量读取；**大蛋白复合物**在 CPU 上可行但耗时大，**尚未**作为生产跑数在本周报周期内完整跑满 10 条先导的 MM-GBSA 全量任务。

## 三、REINVENT4 子进程问题：原因与修复范围

### 3.1 现象（3 月 20 日日志）

- `reinvent` 返回非 0；日志中曾截断 traceback，易误判为「单一原因」。

### 3.2 已定位的两类问题

1. **动态库**：子进程未优先使用 conda 的 `libstdc++`，导致 RDKit 报 `GLIBCXX_3.4.31` 缺失。修复：在 `reinvent_bridge.py` 的 `subprocess.run` 中注入 `LD_LIBRARY_PATH`（conda env 的 `lib` 前置）。
2. **配置格式**：桥接生成的 TOML 与 **REINVENT 4.7.x** 的 schema 不一致（例如 `run_type` 位置、`[[stage]]`、评分组件结构、`TanimotoSimilarity` 的 `use_features` 等）。修复：按官方示例结构调整模板；ExternalProcess 评分脚本改为符合插件期望的 **JSON payload** 输出。

### 3.3 「RL 训练循环已验证」的准确表述

| 验证项 | 状态 |
|---|---|
| `reinvent --help` / 进程能 import RDKit 并进入主程序 | **已验证**（在注入 `LD_LIBRARY_PATH` 后） |
| 配置通过 Pydantic，进入 `staged_learning` 并创建输出 CSV 前缀文件 | **已验证**（短跑、CPU、限时场景下） |
| **完整 `max_steps`、生成稳定可读的 `reinvent4_1.csv` 分子列表、并接回 T2Lead 下游** | **桥接模块的生产参数全链路**：仍待单独验收；**极简 smoke**（见附录 A）已在本机 **exit 0** 跑完并写出 `reinvent4_1.csv`（含表头 + 多行 step 记录）。 |

**结论**：可表述为 **「在注入 conda `LD_LIBRARY_PATH` + REINVENT 4.7.x 格式 TOML 的前提下，CPU 上可完成短 RL 跑数并落盘 CSV」**；生产级 `n_steps`/batch 与 ExternalProcess 评分仍建议再跑一轮计时与结果抽检。

## 附录 A：REINVENT4 可复现短测（本环境已执行）

**目的**：验证 `reinvent` 子进程在 **conda 的 `libstdc++` 优先** 的前提下能启动 REINVENT 4.7.x，并在 **CPU** 上跑完 **4 个 RL step** 后正常退出，并生成 `reinvent4_1.csv`。

**环境**：`conda` 环境名 `t2lead`；`reinvent` 可执行文件路径以本机为准（示例：`/root/miniconda3/envs/t2lead/bin/reinvent`）；prior 文件需存在（示例：`/root/REINVENT4/priors/reinvent.prior`）。

### A.1 仅验证 CLI（约 1 分钟内首次 import 较慢）

```bash
export LD_LIBRARY_PATH="/root/miniconda3/envs/t2lead/lib:${LD_LIBRARY_PATH:-}"
/root/miniconda3/envs/t2lead/bin/reinvent --help
```

### A.1b（可选）去掉 prior 里无效 metadata，避免启动时刷 ERROR

若 prior 为自定义训练产物且含 `hash_id` 与文件不一致，REINVENT 会打 ERROR 大段日志。可先备份并去掉顶层 `metadata`：

```bash
conda run -n t2lead python /root/Projects/T2Lead/scripts/fix_reinvent_prior_metadata.py /root/REINVENT4/priors/reinvent.prior
```

### A.2 极简 RL 跑数（CPU，`max_steps=4`，约 8–10 分钟）

**推荐**：`--log-level error` 只保留 ERROR 及以上，屏蔽 INFO/WARN（含「Inception」「use_cuda deprecated」等）。T2Lead 流水线里 `hit_to_lead.reinvent4.log_level` 默认已设为 `error`。

```bash
mkdir -p /tmp/reinvent_smoke
export LD_LIBRARY_PATH="/root/miniconda3/envs/t2lead/lib:${LD_LIBRARY_PATH:-}"
/root/miniconda3/envs/t2lead/bin/reinvent --log-level error -d cpu /root/Projects/T2Lead/scripts/reinvent_smoke.toml
```

**说明**：仓库内同内容配置见 [scripts/reinvent_smoke.toml](../../scripts/reinvent_smoke.toml)；其中 `summary_csv_prefix` 与 `chkpt_file` 指向 `/tmp/reinvent_smoke/`，若需改路径请同步修改 TOML。
