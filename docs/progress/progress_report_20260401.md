# T2Lead 项目进展与规划汇报

> 2026-04-01 · 阶段性总结

---

## 一、项目定位

T2Lead 是一套端到端的小分子药物发现引擎：输入一个疾病名称，自动完成靶点发现、虚拟筛选、先导化合物生成与优化，输出可供实验验证的候选分子及其完整评估数据。

核心设计原则：**单命令全自动、全链路可复现、每个环节可独立替换**。

---

## 二、已实现架构：四阶段主流水线

```
  疾病名称
     │
     ▼
┌──────────────────────────────────────────────────────────────────┐
│  Stage 1 · 靶点发现                                              │
│  OpenTargets GraphQL → 疾病关联靶点排序                           │
│  自动选择 ChEMBL IC50 数据量充足的靶点（而非盲选评分最高者）        │
└──────────────┬───────────────────────────────────────────────────┘
               │ Target ChEMBL ID
               ▼
┌──────────────────────────────────────────────────────────────────┐
│  Stage 2 · 靶点到苗头 (Target → Hit)                             │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────────────────┐  │
│  │ ChEMBL 数据  │→│ RF + MLP 集成 │→│ 285万分子库虚拟筛选     │  │
│  │ 抓取与清洗   │  │ pIC50 预测模型│  │ + QED/PAINS/ADMET 过滤 │  │
│  └─────────────┘  └──────────────┘  └────────────────────────┘  │

┌──────────────────────────────────────────────────────────────────┐

│  ┌──────────────────────────────────┐                            │
│  │ REINVENT4 强化学习分子生成 (12000+)│ ← AstraZeneca 开源       │
│  └──────────────────────────────────┘                            │

               │ Lead SMILES + MPO score
               ▼
┌──────────────────────────────────────────────────────────────────┐
│  Stage 4 · 先导优化 (Lead Optimization)                          │
│                                                                  │
│  蛋白准备 ─→ AutoDock Vina 对接 ─→ 增强 ADMET ─→ MM-GBSA (GPU)  │
│  (PDBFixer)   (exhaustiveness=32)   (hERG/CYP/    (OpenMM CUDA) │
│                                      Veber/SA)                   │
│                       │                                          │
│                       ▼                                          │
│              Composite Scoring:                                  │
│              w_dock × dock + w_md × energy + w_stab × RMSD       │

│                                                                  │

└──────────────────────────────────────────────────────────────────┘
```

**工程能力指标：**

| 指标 | 数值 |
|------|------|
| 单次全流程耗时（含 MD） | ~9 小时（GPU），缓存重跑 <10 秒 |
| 虚拟筛选库规模 | 285 万分子 |
| REINVENT4 RL 新生成分子 | 12,386 个有效唯一结构 |
| 模型预测精度 | RF R²=0.77, CV R²=0.73±0.02 |
| 对接成功率 | 50/50 (100%) |
| ADMET 通过率 | 10/10 全部通过 hERG/CYP/Veber |

**创新点 ①：智能靶点选择策略**

不盲选关联评分最高的靶点——BRCA1 虽然遗传学证据评分最高（0.909），但 ChEMBL 仅 20 条 IC50 数据，无法训练可靠模型。系统自动切换到数据充足的 PIK3CA（9,728 条），保证了下游 ML 模型的统计效力。这一逻辑在其他公开流水线中未见。

**创新点 ②：RL 生成 × 传统筛选的融合架构**

Stage 3 同时运行 CREM 骨架改造（传统化学空间搜索）和 REINVENT4 强化学习（目标导向生成），两条路径的分子回流后统一 MPO 排序，兼顾化学空间覆盖度和目标导向性。

**创新点 ③：权重自适应 composite scoring**

opt_score 的各维度权重在组件缺失时自动重归一化（如 MD 失败则 docking + ADMET 自动接管全部权重），保证流水线在任何硬件配置下都能给出有意义的排序，而非整体失败。

## 三、正在推进：外显子到小分子配体的闭环

### 目标

实现"患者肿瘤样本 → 体细胞突变 → 突变蛋白结构 → 针对性分子筛选"的全自动闭环。

### 架构

```
 肿瘤 FASTQ + 正常 FASTQ
          │
          ▼
 ┌─────────────────────┐
 │  nf-core/sarek       │  BWA-MEM 比对 → Mutect2 体细胞突变检测
 │  (Nextflow 容器化)   │  → VEP 变异注释
 └─────────┬───────────┘
           │ 注释后 VCF
           ▼
 ┌─────────────────────┐
 │  VCF Parser          │  提取蛋白层变异：EGFR L858R, PIK3CA H1047R ...
 │  (支持 VEP/SnpEff)  │  过滤 MODERATE + HIGH impact
 └─────────┬───────────┘
           │ SomaticVariant 列表
           ▼
 ┌─────────────────────┐
 │  Mutant Sequence     │  UniProt/Ensembl 获取野生型序列
 │  Builder             │  → 应用氨基酸替换 → 输出突变 FASTA
 └─────────┬───────────┘
           │ 突变蛋白序列
           ▼
 ┌─────────────────────┐
 │  Structure Bridge    │  决策树：
 │                      │  ① RCSB 有实验结构？→ 下载 + 点突变
 │                      │  ② 无 → ESMFold/Boltz-2 预测突变结构
 └─────────┬───────────┘
           │ 突变蛋白 PDB
           ▼
 ┌─────────────────────┐
 │  Stage 4 (复用)      │  对每个突变体独立运行：
 │  对接 + MD + 排序    │  docking → ADMET → MD → opt_score
 │                      │  输出路径: stage4_optimization/<GENE_MUT>/
 └──────────────────────┘
```

**当前状态：**

| 模块 | 状态 | 说明 |
|------|------|------|
| SarekRunner | 代码完成 | Nextflow 调用封装，支持 Singularity/Docker |
| VCF Parser | 代码完成 | 内置轻量解析，可选 pysam/cyvcf2 加速 |
| Mutant Sequence Builder | 代码完成 | UniProt + Ensembl 双源序列获取 |
| Structure Bridge | 代码完成 | RCSB 实验结构优先 → ESMFold 兜底 |
| Pipeline 集成 | 代码完成 | `variant_analysis.enabled: true` 触发，自动循环所有突变体 |
| 端到端验证 | 待进行 | 需要真实肿瘤/正常 FASTQ 数据做首次全链路测试 |

**创新点 ④：从基因型到候选药物的全自动桥接**

目前绝大多数药物发现流水线止步于"给定靶点 → 筛选分子"。T2Lead 将上游的体细胞突变检测（nf-core/sarek）与下游的结构预测 + 对接评估打通：一个 VCF 文件进去，出来的是"这个患者的突变蛋白，哪些候选分子结合最好"的排序表。这在同类开源工具中是独有的。

**创新点 ⑤：结构预测策略自动路由**

Structure Bridge 实现了多级后备策略：实验结构 → ESMFold API → 本地 ESMFold → ColabFold，自动选最优可用路径，避免因单一结构来源失败而中断流程。后续 Boltz-2 / Chai-1 作为可插拔后端接入。

## 四、规划方向 A：Benchmark 评分模块

### 解决的问题

当前流水线只能评估自己生成的分子。无法对已有药物（如 Alpelisib）跑同一套评分，也就无法直接回答"我们的候选比已上市药好在哪"。

### 架构设计

```
输入: SMILES 列表 (已有药物 / 竞品分子 / 文献化合物)
       │
       ▼
 ┌──────────────────────────────────────────────────────────┐
 │  BenchmarkScorer                                         │
 │                                                          │
 │  复用已有模块，不写新算法：                                │
 │  ├── utils/chem.py       → MW, LogP, QED, SA, TPSA     │
 │  ├── model_cache/        → pIC50 预测 (RF+MLP ensemble) │
 │  ├── hit_to_lead/mpo.py  → MPO score                    │
 │  ├── docking.py          → Vina 对接 (复用已有 receptor) │
 │  ├── admet_deep.py       → hERG, CYP, Veber, SA        │
 │  └── lead_optimizer.py   → opt_score (composite)        │
 │                                                          │
 │  可选: MD 模拟 (耗时长，默认跳过)                         │
 └──────────────────────────────────────────────────────────┘
       │
       ▼
输出: 与 optimized_leads.csv 完全一致的列
      → 直接拼接对比
```

**使用场景：**

```bash
# 对比 Alpelisib 和 Inavolisib
t2lead benchmark \
  --smiles-csv reference_drugs.csv \
  --run-root /autodl-fs/T2Lead/breast_cancer \
  --skip-md
```

**工作量预估：** 2-3 天。所有子评分函数已经存在，只需要一个新入口做拼装。

**核心价值：** 每次跑完流水线，自动附上"vs 同靶点已上市药"的 head-to-head 对比表。这是向投资人、临床合作方展示价值最直接的方式。

## 五、规划方向 B：可视化报告生成

### 解决的问题

当前输出只有 CSV + 日志文件，非技术人员无法快速理解结果。

### 架构设计

在 Stage 4 结束后自动生成交互式 HTML 报告：

```
stage4_optimization/
├── optimized_leads.csv          ← 已有
├── docking_poses/               ← 已有
└── report.html                  ← 新增：浏览器打开即可
```

**报告内容：**

| 板块 | 内容 |
|------|------|
| 总览卡片 | 候选数、通过率、最佳 opt_score、靶点信息 |
| 排名表 | 交互式排序/筛选，点击展开分子详情 |
| 气泡图 | pIC50 vs Docking (大小=opt_score, 颜色=RMSD) |
| 雷达图 | Top N 候选分子多维度对比 |
| ADMET 热图 | hERG / CYP / Veber / SA 各项通过情况 |
| 2D 结构图 | RDKit 渲染的分子结构 |
| Benchmark 对比 | 若有参考药物，展示 head-to-head 对比 |

**技术方案：** Plotly 生成图表 + Jinja2 模板 → 单个自包含 HTML，无需服务器，离线可查看。

**工作量预估：** 1-2 天。

## 六、规划方向 C：患者个体化评分加权

### 核心思路

将患者自身的生物学特征（代谢酶分型、组学数据等）转化为评分权重调节因子，使同一批候选分子对不同患者产生不同的优先级排序。

### 三个实施层次（供选择）

---

#### 层次 1 · 代谢酶分型驱动（推荐起步）

```
输入:  患者 CYP 基因分型报告
       (CYP2D6 慢代谢 / 中间型 / 正常 / 超快代谢)
       ──────────────────────────────────────
         ↓  查表映射
       ──────────────────────────────────────
中间:  代谢风险系数 α_cyp ∈ [0.3, 1.5]

       opt_score 中 w_cyp_soft 动态调整:
       慢代谢者 → 提高 CYP 权重 (更惩罚 CYP 底物)
       超快代谢者 → 降低 CYP 权重 (CYP 底物反而代谢快)
       ──────────────────────────────────────
输出:  个性化 opt_score → 不同患者看到不同排序
```

| 维度 | 说明 |
|------|------|
| **输入** | CYP 基因分型（临床已有成熟检测，PGx 标准化） |
| **模型** | 无需训练，查表映射（PharmGKB 数据库） |
| **难点** | 低，主要是工程拼接 |
| **改动范围** | config.yaml 增加 `patient_profile` 段，`_composite_score` 改为动态权重 |
| **工作量** | 1-2 天 |

---

#### 层次 2 · 多组学小模型加权

```
输入:  多维度患者数据
       ┌──────────────────┐  ┌──────────────────┐  ┌────────────────┐
       │ 舌苔代谢组       │  │ 肠道菌群 16S     │  │ 血液转录组     │
       │ (LC-MS 特征)     │  │ (OTU 丰度表)     │  │ (RNA-seq TPM)  │
       └────────┬─────────┘  └────────┬─────────┘  └───────┬────────┘
                │                     │                     │
                ▼                     ▼                     ▼
       ┌────────────────┐  ┌────────────────┐  ┌────────────────────┐
       │ 炎症状态模型    │  │ CYP 活性模型    │  │ hERG 易感性模型    │
       │ (XGBoost/LR)   │  │ (XGBoost/LR)   │  │ (XGBoost/LR)      │
       └───────┬────────┘  └───────┬────────┘  └──────────┬─────────┘
               │0.82               │0.65                   │0.3
               ▼                   ▼                       ▼
       ┌─────────────────────────────────────────────────────────────┐
       │  权重映射层                                                 │
       │  inflammation_score → w_admet_boost (炎症高→更看重安全性)   │
       │  cyp_activity_score → w_cyp_soft (代谢活性→调 CYP 权重)    │
       │  cardiac_risk_score → w_herg_penalty (心脏风险→惩罚 hERG)  │
       └─────────────────────────────────────────────────────────────┘
               │
               ▼
       个性化 opt_score
```

| 维度 | 说明 |
|------|------|
| **输入** | LC-MS 代谢组学、16S 扩增子、RNA-seq 等（需标准化采集方案） |
| **模型** | 每个数据源独立训练一个轻量分类/回归模型，输出 0-1 分数 |
| **难点** | ① 各组学数据标准化（特别是舌苔等非常规数据源）<br>② 每个模型需要足够的训练-验证队列<br>③ 模型输出到权重的映射需要临床可解释性 |
| **改动范围** | 新增 `src/drugpipe/patient_profile/` 模块 + 配置段 |
| **工作量** | 每个数据源 1-2 周（含数据收集与模型验证） |

---

#### 层次 3 · 端到端个性化药物响应预测

```
输入:   患者多组学特征向量 + 候选分子指纹
            │
            ▼
     ┌─────────────────────────────────┐
     │  深度学习模型                    │
     │  (Drug-Patient interaction)      │
     │  类似 DeepCDR / TGSA 架构       │
     │  患者编码器 × 分子编码器 → 响应  │
     └────────────────┬────────────────┘
                      │
                      ▼
     预测该患者对该分子的响应概率
     → 直接作为排序依据
```

| 维度 | 说明 |
|------|------|
| **输入** | 患者完整组学 profile + 分子 SMILES |
| **模型** | 端到端深度学习，药物-患者交互预测 |
| **难点** | ① 需要大规模患者-药物响应配对数据（公开数据集如 GDSC/CCLE 是细胞系而非真实患者）<br>② 模型可解释性不足，临床落地需要额外工作<br>③ 训练和验证成本高 |
| **改动范围** | 新增独立模型训练流水线 + 推理集成 |
| **工作量** | 3-6 个月研究型开发 |

---

#### 推荐路径

```
层次 1 (立即启动)  ──→  层次 2 (有数据后逐步加)  ──→  层次 3 (研究储备)
   1-2 天                   按数据源逐个迭代               中长期课题
```

**先落层次 1，架构上预留层次 2 的扩展接口。** 原因：层次 1 改动极小但概念完整（"个性化评分"的故事已经能讲），层次 2 的瓶颈在数据而非代码，层次 3 是发论文的方向但不急于产品化。

## 七、技术高地与核心竞争力总结

| # | 能力 | 说明 | 同类工具对比 |
|---|------|------|------------|
| 1 | **疾病名 → 候选分子一键到底** | 四阶段全自动，无需人工介入 | 多数工具只覆盖 1-2 个阶段 |
| 2 | **RL 生成 + 传统筛选融合** | REINVENT4 与 CREM 双路径产出统一排序 | 通常只选其一 |
| 3 | **基因变异到配体的全自动闭环** | FASTQ/VCF → 突变结构 → 对接评估 | 同类开源工具中未见 |
| 4 | **物理验证层（非纯 ML）** | AutoDock Vina + OpenMM MD + MM-GBSA | 多数止步于 ML 预测 |
| 5 | **容错自愈架构** | MD 失败→权重重分配；REINVENT 失败→fallback；靶点数据不足→自动切换 | 通常整段失败 |
| 6 | **三级缓存系统** | 指纹/模型/筛选结果全缓存，迭代开发 <10 秒重跑 | 多数每次从头计算 |
| 7 | **多入口场景支持** | 标准模式 / 外部数据 / 纯对接 / 蛋白序列模式 | 通常只有单一入口 |
