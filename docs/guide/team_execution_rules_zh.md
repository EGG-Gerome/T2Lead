# 团队执行规则（T2Lead）

## 1) 环境与命令

- 项目命令默认使用 `conda` 环境，不使用 `base`。
- 推荐命令形式：`conda run -n <env_name> <command>`。
- 长任务启动前，先确认 Python 与依赖环境一致。

## 2) 长任务运行方式

- 预计超过 10 分钟的任务，默认使用 `tmux` / `screen` / `nohup`。
- 禁止将长任务仅挂在前台 SSH 会话。
- 启动后必须提供进程检查与日志检查命令。

## 3) 医药项目质量要求

- 准确性和可追溯性优先于速度。
- 报告必须区分“观察事实”和“原因推断”。
- 不确定结论必须写置信度说明。

## 4) 代码变更验证

- 非微小改动后必须跑相关测试或 smoke。
- 未验证不宣称“已完成/可用”。
- 推行“小步快跑”：每做完一个小功能/修复，就立刻测试再继续下一个。

## 5) 超大变异集运行策略

- 先小批量冒烟（例如 20-50 变异）验证流程与外部依赖。
- 再按批次执行（例如每批 100-300 变异），便于恢复与追踪。
- 每批都记录输入、命令、日志和输出位置。

## 6) Git 工作方式

- 每个小任务完成后至少执行：
  - `git status`
  - `git diff`（确认改动符合预期）
- 测试通过后再发起提交/PR（避免“未验证提交”）。
- 提交粒度尽量小（单一意图），便于回滚和代码审阅。
- 分支建议：`feat/*`、`fix/*`、`docs/*`、`chore/*`。

## 7) 文档归档规范

- 每次运行统一归档到：
  - `docs/experiments/runs/<date>_<run-name>_<run-id>/`
- 至少包含：
  - `REPORT.md`
  - `ARTIFACTS.md`
  - `COMMANDS.md`
  - `METADATA.yaml`
- 团队内部默认中文，可按协作需求补英文。
